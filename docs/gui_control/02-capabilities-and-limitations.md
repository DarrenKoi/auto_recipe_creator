# GUI 제어의 가능성과 한계

자동화 시스템을 설계할 때 무엇이 가능하고 무엇이 불가능한지(또는 매우 어려운지) 아는 것은 매우 중요합니다.

## 1. 할 수 있는 것 (Capabilities)

1.  **표준 컨트롤 조작:** 버튼 클릭, 메뉴 선택, 체크박스 토글, 텍스트 입력.
2.  **창 관리:** 창 최소화/최대화, 위치 이동, 포커스 전환, 창 닫기.
3.  **데이터 추출:** 화면의 텍스트 읽기 (OCR 또는 객체 속성 추출), 색상 확인.
4.  **복잡한 매크로:** 수백 단계의 반복적인 마우스/키보드 동작을 오차 없이 수행.
5.  **동적 대기:** 특정 조건(파일 생성, 네트워크 응답, UI 변화)이 충족될 때까지 지능적으로 대기.

## 2. 할 수 없는 것 또는 어려운 것 (Limitations)

### 2.1 보안 및 권한 제약 (UAC)
- **문제:** 관리자 권한으로 실행된 프로그램(예: Task Manager, 일부 설치 프로그램) 위에서는 일반 권한의 자동화 스크립트가 동작하지 않습니다.
- **해결:** 스크립트 자체를 관리자 권한으로 실행해야 합니다.

### 2.2 하드웨어 수준의 보호
- **문제:** 일부 보안 소프트웨어나 게임(Anti-cheat)은 소프트웨어적인 가상 입력을 차단합니다.
- **해결:** 커널 레벨 드라이버를 사용하거나, USB 마우스 신호를 가로채는 하드웨어(Arduino/Raspberry Pi)를 사용해야 합니다.

### 2.3 비표준 UI (Custom Graphics)
- **문제:** DirectX나 OpenGL로 그려진 화면(게임, 특수 산업용 뷰어)은 Windows가 객체로 인식하지 못합니다.
- **해결:** 이 경우 객체 기반 제어가 불가능하며, **이미지 매칭**이나 **VLM 분석**을 통한 좌표 기반 제어만 가능합니다.

### 2.4 네트워크 지연 (RCS 환경)
- **문제:** 원격 제어 솔루션(RCS, VNC)을 통해 제어할 경우, 로컬 화면과 실제 장비 화면 사이에 시차가 발생합니다.
- **해결:** 입력을 보낸 후 화면이 바뀔 때까지 기다리는 **피드백 루프** 설계가 필수적입니다.

### 2.5 예외 상황 대응의 한계
- **문제:** 갑작스러운 팝업 광고, 윈도우 업데이트 알림, 네트워크 끊김 등 예측하지 못한 UI 변화.
- **해결:** 모든 예외를 코드로 짜는 것은 불가능하므로, 에러 발생 시 사람에게 알림을 주는 **Supervised Autonomy** 모델이 권장됩니다.
